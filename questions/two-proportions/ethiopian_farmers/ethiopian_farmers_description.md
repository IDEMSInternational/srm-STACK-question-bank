GOAL: Test a student's ability to carry the same 2×2 table through two parallel inferential frameworks — a two-proportion z-test/CI and a chi-square-based odds-ratio test/CI — including recognizing which quantities each framework compares, computing each, drawing a conclusion, and checking the validity conditions.

DESCRIPTION: The student sees a 2×2 table of adopters/non-adopters for small vs. large firms, then answers 12 parts across three blocks. Thread 1 (proportions): an MCQ identifying how the compared quantity is defined, a two-number box for the pair of sample proportions, an algebraic box for the z-statistic, an MCQ decision (reject/fail-to-reject), and a two-number box for a confidence interval. Thread 2 (odds ratio) mirrors this exactly: MCQ for the OR definition, a two-number box for the odds pair, an algebraic box for the chi-square statistic, an MCQ decision, and a two-number CI box. A final Validity block asks for the smallest expected cell count (numeric) and an MCQ (valid/invalid vs. the ≥5 rule).

STRUCTURE:
1. Thread 1 — proportion difference (5 inputs: ans1a–ans1e), each with its own PRT: (a) MCQ definition, graded directly; (b) two-number p̂ pair, graded against true values; (c) z-statistic, recomputed from the student's own ans1b pair (pooled SE) and compared to the student's answer; (d) decision MCQ, re-derived from the student's own ans1c value vs ±1.96; (e) CI, recomputed from the student's own ans1b pair (unpooled SE).
2. Thread 2 — odds ratio (5 inputs: ans2a–ans2e), structurally identical to Thread 1: MCQ definition; odds pair; χ² statistic follow-through from the student's own ans2b (reconstructing an implied 2×2 table from the odds and each row's total); decision MCQ follow-through from the student's own ans2c vs 3.84; OR-CI follow-through from the student's own ans2b.
3. Validity (2 inputs: ans3a, ans3b): smallest expected cell count, graded directly; valid/invalid MCQ, follow-through from the student's own ans3a vs the ≥5 threshold.
Each of the 12 inputs has its own independent PRT; there is no cross-thread dependency, matching the plan exactly as built.

RANDOMIZATION: A regime flag (0/1, equally likely) selects between two generation paths. Regime 0 ("normal"): row totals rand(80)+80 each, base proportions giving moderate-to-large samples; typically produces valid expected counts and a mix of reject/fail-to-reject outcomes across variants. Regime 1 ("small-N"): row totals rand(15)+10 each with base proportions rand(40)+10 over 100, deliberately prone to producing an expected cell count under 5 so the validity check isn't trivially always "valid." Cell counts A–D are derived from the row totals and proportions (rounded), with `s_assert` guarding that all four cells are strictly positive so no variant produces a degenerate table (a zero cell, which would break odds/log-odds calculations).

ANSWER TESTS: MCQs (ans1a/ans2a definitions) graded directly against the correct option; decision and validity MCQs graded via a feedback-variable-computed "expected option" derived from the student's own upstream numeric answer, then compared with AlgEquiv/`is(...)`. Two-number inputs (p̂ pair, odds pair) are algebraic list inputs graded via a custom AlgEquiv-wrapped boolean check with fixed absolute tolerances (0.001 for p̂, 0.01 for odds) against the true values. The z-statistic, χ² statistic, and both CIs are algebraic inputs graded by a feedback-variable block that reconstructs the "correct" value from the student's own upstream two-number answer (not the teacher's true values) and compares within a tolerance (0.01 for z/CI-diff, 0.05 for χ²/OR-CI). The expected-count input is graded directly against the true minimum expected count (tolerance 0.01).

FEEDBACK: Each downstream numeric/decision node's feedback explicitly states it is using the student's own prior answer, not the true table values, so a discrepancy's source is traceable. The general feedback presents both threads side-by-side against the true table, explicitly distinguishes pooled SE (used for z) from unpooled SE (used for the proportion CI), and makes an explicit teaching point that the two hypothesis-test decisions must always agree because χ² = z² exactly (1.96² = 3.8416 ≈ 3.84), while also noting the CI-based read can occasionally diverge near a boundary from the test-statistic decision because CIs use unpooled SE.

IMPLEMENTATION NOTES:
- Odds-pair follow-through (ans2c/ans2e) works by algebraically reconstructing an implied 2×2 table (A,B,C,D) from the student's own odds pair and the true row totals n_s/n_l, then recomputing χ² or the OR-CI from that reconstructed table — this is the load-bearing trick making odds-ratio follow-through mathematically well-defined.
- Guard clauses throughout (e.g. `if not(listp(...) and length(...)=2) then false else ...`, and division-by-zero guards on se/chisq/selog) mean a malformed or degenerate upstream answer (e.g. an empty list) fails cleanly as "false" and does not propagate a Maxima runtime error into downstream PRTs — verified in qtest case 5.
- Reviewer's PRT/QTEST guidance not to test for disagreement between the two threads' hypothesis-test decisions (since χ²=z² makes disagreement mathematically impossible) was followed — no such case appears in the qtests, and this equivalence is called out directly in general feedback as an intentional teaching point rather than a coincidence.
- The suggested "CI vs test-decision boundary divergence" adversarial case (from PRT/QTEST SUGGESTIONS) was not built into a qtest case — the six qtests actually present focus on definition-MCQ errors, thread-specific follow-through, an empty-list adversarial input, and validity follow-through instead. A future editor wanting that specific boundary-divergence check will need to add it.
